Strategy Pattern Report — Exercise 6

Project: Website Monitoring System  |  Date: June 17, 2026  |  Pattern: Strategy (Behavioral)

1. Goal

Exercise 5 introduced the Observer pattern so that a Website notifies its subscribers automatically when its content changes. Change-detection, however, was hard-coded as a single exact-string comparison inside updateContent(): 

if (!currentContent.equals(cachedContent)) {
    notifyObservers();
}

Exercise 6 requires the system to support multiple comparison strategies for deciding when a website has "changed":

  1. Identical content size — change only when the length differs.
  2. Identical HTML content — change on any raw-HTML difference.
  3. Identical text content — change only when the visible text differs (markup-only edits are ignored).

Hard-coding three different if branches would violate the Open/Closed Principle and force Website to change every time a new rule is added. The Strategy pattern solves this by extracting the comparison rule into interchangeable objects.

2. Strategy Pattern Solution

The comparison algorithm is encapsulated behind a ComparisonStrategy interface. Website becomes the Context: it holds a reference to a strategy and delegates change-detection to it, without knowing which concrete rule is in use. The strategy can be swapped at runtime through setComparisonStrategy().

4. Key Code

4.1 The Strategy interface

public interface ComparisonStrategy {
    // true  -> contents are identical (no change, no notification)
    // false -> contents differ      (change -> notify observers)
    boolean isIdentical(String oldContent, String newContent);
    String getName();
}

4.2 The three concrete strategies

// 1. Identical content size
public boolean isIdentical(String oldContent, String newContent) {
    return oldContent.length() == newContent.length();
}

// 2. Identical HTML content
public boolean isIdentical(String oldContent, String newContent) {
    return oldContent.equals(newContent);
}

// 3. Identical text content
public boolean isIdentical(String oldContent, String newContent) {
    return extractText(oldContent).equals(extractText(newContent));
}
private String extractText(String html) {
    return html.replaceAll("<[^>]*>", " ")
               .replaceAll("\\s+", " ")
               .trim();
}

4.3 Website as the Context

private ComparisonStrategy comparisonStrategy;

public void setComparisonStrategy(ComparisonStrategy strategy) {
    this.comparisonStrategy = strategy;          // swap rule at runtime
}

public void updateContent(String contentUpdated) {
    this.currentContent = contentUpdated;
    if (!comparisonStrategy.isIdentical(cachedContent, currentContent)) {
        notifyObservers();                       // delegated decision
        this.cachedContent = currentContent;
    }
}

6. Coding Conventions

The following conventions were applied consistently across the codebase (Exercise 6, task 3):

AreaConvention used
Naming — classesPascalCase, nouns describing the role (Website, ComparisonStrategy, ContentSizeComparison).
Naming — interfacesNamed by capability/role rather than an I prefix (Subject, Observer, ComparisonStrategy), following standard Java style.
Naming — methods & fieldscamelCase; methods are verbs (updateContent, isIdentical, notifyObservers); boolean queries use an is prefix.
Naming — constants / enumsUPPER_SNAKE_CASE enum constants (FREQUENCY.DAILY, SubscriptionStatus.ACTIVE).
EncapsulationFields are private; access is through getters/setters. Collaborators are passed in via constructors or setters (dependency injection of the strategy).
Class layoutFields first, then constructors, then public methods, then helpers, then toString(). One top-level class per file (file name matches the public class).
Indentation & braces4-space indentation, K&R brace style (opening brace on the same line), consistent throughout.
CommentsJavadoc /** ... */ on public types and non-trivial methods; short inline // comments explain intent, not mechanics.
OverridesAll interface/inherited implementations are marked with @Override.
Single ResponsibilityEach class has one job; new behaviour is added by writing a new class (Open/Closed), not by editing existing ones.
DefaultsWebsite provides a default strategy (HtmlContentComparison) via constructor chaining so existing call sites keep working.
